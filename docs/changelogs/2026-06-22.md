## 第三部分：ST 样式修复 (2026-06-22 晚间)

### ST 页面无排版问题
- **现象**：`http://localhost:8080/st/` 页面无任何 CSS 样式，裸 HTML
- **根因**：ST 的 JavaScript 在页面加载后把 `<base href="/st/">` 重置为 `<base href="/">`，导致所有 CSS/JS 从 `/css/...` 加载 → nginx 把这些请求路由到 Dashboard → 404
- **修复**：放弃 sub-path 代理方案，给 ST 分配独立端口 `8082`。nginx 新增独立 server block 直连 ST，无需路径前缀重写。
- **变更**：
  - `nginx.conf` — 移除 `/st/` location，新增 `listen 8082` server block
  - `docker-compose.yml` — nginx 新增 `8082:8082` 端口映射
  - `BridgePage.tsx` — 聊天按钮改为 `localhost:8082`

### 访问地址
| 地址 | 用途 |
|------|------|
| `http://localhost:8080/` | Dashboard + BridgePage |
| `http://localhost:8082/` | SillyTavern Web UI |

## 第四部分：WebSocket 双向通信 (2026-06-22 深夜)

### 架构修正
- **问题1**：ChatBridge 扩展是 WS 客户端，relay 原来也是 WS 客户端，两个客户端无法通信
- **问题2**：ChatBridge 只处理接收消息，不发送 AI 回复回传
- **修复**：
  - `hermes-st-relay.js` — 改为 WebSocket **Server**，监听 `:8850`
  - `ChatBridge index.js` — 加 identify 握手 + `GENERATION_ENDED` 事件捕获 + 回复回传

### 连接方式
```
ST ChatBridge (客户端) → ws://172.18.0.1:8850 → hermes-st-relay (服务端)
```

### ST 中配置 ChatBridge
- **重要**：ChatBridge JS 运行在**浏览器**中（不是 Docker），必须打开 ST 页面才会连接
- **关键问题**：ST 从 `public/scripts/extensions/third-party/` 加载扩展，不是 `data/` 目录
- **修复**：
  - `docker cp` 更新文件到容器内正确路径
  - `docker-compose.st.yml` 加 bind mount 确保持久化
  - 默认地址 `172.18.0.1:8850` + `autoConnect: true`
- **验证**：打开 `http://localhost:8082/`，扩展面板显示"已连接"，relay 日志显示 `ST client registered`
- **注意**：`172.18.0.1` 是 Docker bridge gateway，从浏览器（宿主机）能路由回宿主机的 relay 端口
